### 1.2 本项目领先的地方

1. **自动补全 8种**（核心优势）— 标题自动空格、加粗自动闭合、链接模板、代码块闭合、代码内智能回车、任务列表 toggle、智能退格配对删除、表格分隔行自动补全。大多数编辑器只做快捷键，不做智能补全。**MarkText 有基础 auto-pair 但远不及此。**

2. **语法 Lint 7类+一键修复**（独有）— 未闭合代码块检测、链接完整性检查、标题格式规范（层级跳跃/#后空格/空标题）、列表格式规范、未闭合行内代码检测、图片完整性与替代文字检查、水平分割线字符统一。MarkText/bytemd/Milkdown 全都没有。

3. **表格可视化编辑器** — 独立的 TableEditor 组件，可视化操作 Markdown 表格，主流编辑器大多只能手写。

4. **博客生成器 9主题**（独有）— 所有开源 Markdown 编辑器中唯一内置博客生成的。

### 1.3 本项目弱于主流方案的地方

1. **WYSIWYG 非原生** — 见下节详细分析。

2. **没有插件系统** — 所有功能硬编码，无法扩展。Milkdown 和 Toast UI Editor 有完整插件架构。

3. **没有移动端适配** — 分屏布局在手机上不可用。

4. **PDF 导出质量差** — 依赖 html2pdf.js（已知不稳定的库）。

5. **许可证未声明** — 不能合法商用，需要联系作者授权。

### 1.4 重要澄清：本项目到底有没有 WYSIWYG？

**有，但不是真正的 WYSIWYG。** 它的预览区右上角有一个"编辑"按钮，点击后预览区变成 `contentEditable`，可以直接在渲染后的 HTML 上编辑，然后通过 **TurndownService** 把修改后的 HTML 反向转回 Markdown。

**它的 WYSIWYG 链条（Round-trip 架构）：**

```
你打字 → textarea（源码编辑区）
         ↓ marked 解析
         HTML → 渲染在右侧预览区
                  ↓ 用户点击"编辑"
         contentEditable=true（编辑 HTML）
                  ↓
         TurndownService → HTML → Markdown（有损转换）
                  ↓
         塞回 textarea → 重新渲染
```

**MarkText/Typora 的 WYSIWYG（AST-native 架构）：**

```
你打字 → 直接修改代码语法树（AST）
         ↓
         渲染器 → 渲染为带样式的界面
         ↓
         没有"源码"和"预览"的分离，看不到 Markdown 符号
```

**实际区别：**

| 场景 | 本项目 | MarkText |
|------|--------|----------|
| 写 `**加粗**` | 左侧 textarea 写 `**加粗**`，右侧预览显示加粗 | 直接显示加粗，永远看不到 `**` |
| 写 `# 标题` | 左侧行首有 `#`，右侧渲染成大标题 | 直接渲染成大标题样式 |
| 切编辑模式 | 要点"编辑"按钮进入 contentEditable | 不需要切，本身就是 WYSIWYG |
| 数据回传 | HTML→Turndown→Markdown（有损） | AST→序列化→Markdown（无损） |

**有损转换举例：** 你在预览区把一段文字改成红色，TurndownService 转回 Markdown 时，**Markdown 没有颜色概念**，这个改动会丢失。

结论：**本项目有"contentEditable 预览编辑"功能，可以算一种简陋的 WYSIWYG。但它不是真正的 WYSIWYG——真正的 WYSIWYG 建立在语法树上，不是建立在做 HTML 往返转换上。**

### 1.5 市场定位结论

**功能完整度超过大部分开源编辑器组件，但不如成熟的独立编辑器应用。在"编辑器+博客工作室"这个交叉领域没有对标开源项目。**

## 二、8 种自动补全详解（核心优势）

全部在 `markdownAutocomplete.ts` 中实现，每个都是独立导出的纯函数。

| # | 功能 | 函数名 | 你做什么 | 它做什么 |
|---|------|--------|---------|---------|
| 1 | 标题自动空格 | `handleHeadingAutoSpace` | 输入 `##内容` | 自动变成 `## 内容` |
| 2 | 加粗自动闭合 | `handleBoldAutoClose` | 输入 `**` | 自动变成 `****`，光标留在中间 |
| 3 | 链接自动补全 | `handleLinkAutoComplete` | 输入 `[` | 自动变成 `[](url)`，光标在中间 |
| 4 | 代码块自动闭合 | `handleCodeFenceAutoComplete` | 输入 ` ```javascript` | 自动补 ` ``` ` 闭合 |
| 5 | 代码块内智能回车 | `handleSmartEnterInCodeBlock` | 在代码块内按回车 | 确保不跳出代码块 |
| 6 | 任务列表切换 | `handleTaskToggle` | 在 `- [ ]` 行上操作 | `- [ ]` ↔ `- [x]` 来回切 |
| 7 | 智能退格 | `handleSmartBackspace` | 在 `**粗体|**` 按退格 | 整对删除 → `粗体` |
| 8 | 表格自动补全 | `handleTableAutoComplete` | 写 `\| 姓名 \| 年龄 \|` + 回车 | 自动补 `\|---\|---\|` 分隔行 |

**对比 MarkText：** MarkText 有 auto-pair（通用符号配对，类似 IDE 的括号自动补），但只做基础，不做表格、代码块、链接框架、智能退格、任务 toggle。

## 三、7 类 Lint 详解（独有优势）

全部在 `markdownLinter.ts` 中实现，纯函数 + 可一键修复。

| # | 规则 | 检查内容 | 示例 | 能否自动修复 |
|---|------|---------|------|------------|
| 1 | 未闭合代码块 | ` ``` ` 开始没有结束 | 整段代码被渲染为普通文本 | ✅ 自动加 ` ``` ` |
| 2 | 链接完整性 | 空文字 `[]()`、空链接 `[]() `、缺括号、孤立方括号 | 断链无法点击 | ✅ 部分修复 |
| 3 | 标题规范 | 层级跳跃、#后缺空格、标题为空 | `##内容` → 提示加空格 | ✅ 加空格 |
| 4 | 列表格式 | 标记后缺空格、混用空格和 Tab | `-项目` → 提示变 `- 项目` | ✅ 加空格 |
| 5 | 未闭合行内代码 | 反引号数量为奇数 | `` ` `` 没有闭合 | ✅ 自动补 |
| 6 | 图片完整性 | 缺括号、缺 Alt 文字 | `![](url)` 不能被屏幕阅读器识别 | ✅ 部分修复 |
| 7 | 水平分割线 | 混用 `*-_` 不同字符 | `*-_-` → 提示统一 | ✅ 统一为 `---` |

**MarkText：完全没有 Lint。**

## 四、博客模块架构分析

### 4.1 整体架构

7 个文件，约 8000 行代码。不是传统 SSG，而是一个**"博客设计工作室"**。

```
BlogStudio.tsx (1330行)        ← UI：可视化设计工作台
    │
    ├── blogGenerator.ts (2187行)   ← 核心引擎
    │   ├── generateBlog()          → 多文件站点
    │   ├── generateSingleFileBlog()→ SPA 单文件
    │   └── downloadBlogAsZip()     → 输出
    │
    ├── blogTemplates.ts (730行)    ← 模板：3布局×5配色×3字体
    │   └── generatePost/Index/Archive/TagsHtml + RSS + Sitemap
    │
    ├── blogThemeStyles.ts (3279行) ← 6 种主题完整 CSS
    │   └── 水墨/终端/报纸/玻璃拟态/便当/叙事
    │
    ├── frontMatter.ts (121行)      ← Front Matter 解析/生成
    ├── FrontMatterPanel.tsx (300行)← 元数据编辑面板
    ├── BlogModal.tsx (778行)       ← 博客预览/设置弹窗
    └── index.ts (7行)
```

### 4.2 技术特点

**SPA 架构：** 所有页面在一个 HTML 中，通过 `display: none/block` 切换 + hash 路由导航。

**纯字符串模板：** HTML 全部通过模板字符串拼接，零依赖但维护困难。

**智能元数据推断：** 不需要手写 Front Matter——自动从 `# 标题` 提取标题、从 `## 二级标题` 提取标签、从日期格式推断发布时间、自动生成摘要、计算阅读时间。

**两种输出：** SPA 单文件（一个 .html 包含所有页面）和多文件 ZIP（index/archive/tags/rss/sitemap + posts/{slug}/index.html）。

### 4.3 局限性

- SPA 对 SEO 不友好
- 没有分页
- blogGenerator.ts 2187 行字符串拼接待重构
- 没有部署流程（生成完就完了）
- **与编辑器耦合**（直接 import imageStore 和 marked）

## 五、MarkText 深度对比

### 5.1 MarkText 是什么

[MarkText](https://github.com/marktext/marktext) 是 **MIT 协议** 的开源桌面 Markdown 编辑器（Electron），**可以免费商用**，GitHub 30k+ stars。

### 5.2 双方优劣势终极对比

| 维度 | 本项目 | MarkText |
|------|--------|----------|
| **协议** | ❌ 未声明（不可用） | ✅ MIT（免费商用） |
| **WYSIWYG** | ⚠️ 分屏+contentEditable 往返 | ✅ 原生 AST 架构 |
| **自动补全** | ✅ **8种，绝对领先** | ❌ 仅基础 auto-pair |
| **语法 Lint** | ✅ **7类，独有** | ❌ 没有 |
| **博客生成** | ✅ **9主题，独有** | ❌ 没有 |
| **表格编辑** | ✅ 独立可视化组件 | ⚠️ 有但不如本项目 |
| **数学符号选择器** | ✅ Popover | ❌ |
| **代码块折叠** | ✅ | ❌ |
| **脚注管理** | ✅ 侧边栏 | ❌ |
| **图片占位符** | ✅ 避免 Base64 膨胀 | ❌ |
| **接口化（可嵌入）** | ✅ 可以抽成 React 组件 | ❌ Electron 深度耦合 |
| **桌面体验** | ❌ Web 应用 | ✅ 原生桌面 |
| **工程成熟度** | ⚠️ 原型级别，债务重 | ✅ 30k+ 用户验证 |
| **国际化** | ❌ 仅中文 | ✅ 10+ 语言 |

### 5.3 一句话总结

> **MarkText 是成熟的编辑器；本项目是"编辑器+博客工作室"的原型。** 在"写"这件事上 MarkText 好得多，在"自动补全/Lint/博客生成"上本项目有独有优势。

## 六、许可证与法律风险

### 6.1 本项目许可证

```json
// package.json
"private": true,
// 没有 "license" 字段
// 没有 LICENSE 文件
```

**法律上默认为 All Rights Reserved（保留所有权利）。** 这意味着：
- ❌ 不能复制、修改、分发代码
- ❌ 不能用于商业项目
- ❌ 不能做二次开发或发布 npm 包
- ❌ 严格来说 fork 都需要许可

这大概率是学生没有意识到要加许可证，不是故意限制。**如果你需要用到本项目的代码，必须先联系作者申请授权，建议让作者加 MIT 协议。**

### 6.2 插件系统

**本项目完全没有插件系统。** 所有功能硬编码，无扩展 API、无插件注册机制、无生命周期钩子。Milkdown 和 Toast UI Editor 有完整插件架构，可以按需加载功能。

## 七、你的实际场景：生产博客管理后台的选型建议

### 7.1 你的现状

```
你已有：
  1. Hexo 静态博客（生产环境）
  2. Inbox 软件（本地 AI 工作流 + Markdown 编辑 + 文件管理）
  3. 这个学生的 MarkdownEditor 项目（正在分析）

你想要：
  浏览器写文章 → 点发布 → 自动部署（不用 git push）
```

### 7.2 选项 A：给 Hexo 加管理后台（最直接）

用 **Decap CMS**（原名 Netlify CMS）给 Hexo 加一个 `/admin` 后台。

**配置方式：**

```
你的 Hexo 项目
├── source/
│   └── admin/
│       ├── index.html          ← CMS 入口
│       └── config.yml          ← 配置内容模型和 Git 后端
├── _config.yml                 ← 加 skip_render: admin/**
└── ...
```

**工作流程：**

```
浏览器访问 yourblog.com/admin
         ↓
Decap CMS 后台（Markdown 编辑器 + 预览）
         ↓
点"发布" → Decap 通过 GitHub API 创建 commit
         ↓
GitHub Actions 检测到新 commit
         ↓
Hexo 自动构建 → 部署
```

**为什么不是 Tina CMS？**

Tina CMS 的可视化在线编辑只支持 **Next.js/React 框架**，Hexo 不在它的支持列表中。对 Hugo/Jekyll/Eleventy 也只有基本编辑器。所以对 Hexo 用户来说，Tina 跟 Decap 的体验是一样的——都是基本编辑器，而 **Decap 跟 Hexo 有成熟集成方案**（中文博客有 2025 年写的完整配置教程）。

### 7.3 选项 B：给 Inbox 加发布功能（长远最优）

Inbox 已经具备：Markdown 编辑、文件管理（目录结构）、AI 绘图导入、网页查看。缺的是"发布到博客"。

只需要加两个能力：

```
1. Git 集成能力
   配置博客 GitHub 仓库 → 点"发布"→ 通过 GitHub API 创建 commit
   文件写入 content/posts/xxx.md，自动生成 Front Matter

2. （可选）随时去 Inbox 里复制自动补全和 Lint
   这个项目的 markdownAutocomplete.ts（296行）和 markdownLinter.ts（543行）
   都是纯函数，没有 UI 依赖，可以直接参考实现
```

这样做的好处：
- 不引入新依赖，你完全掌控
- Inbox 从"本地编辑器"升级为"写→管→发"统一入口
- Inbox 的编辑器如果有欠缺，可以从这个项目参考补全和 Lint

### 7.4 选项 C：用 MarkText 当编辑器

如果只是觉得当前编辑器不好用的权宜之计。MarkText 桌面体验好、免费可商用，但它不解决"发布"问题——写完后依然需要手动把文件放到 Hexo 项目里。

### 7.5 这个学生项目在这里的实际角色

它**不是**你要用的工具。它的价值是：

> **"零件"价值。** 把它最强的两个模块（自动补全 8 种 + Lint 7 类）的**设计思路**和**纯函数代码逻辑**，参考移植到你的 Inbox 编辑器里。它的博客生成器对你的场景反而是浪费——你有自己的博客主题了，不需要生成独立 SPA 博客。

### 7.6 选型决策树

```
你想在浏览器里管理 Hexo 博客？
    ├── 想今天就能用 → Decap CMS（2 小时配置）
    │
    ├── 想让 Inbox 统一管理一切
    │   └── Inbox + GitHub API 发布按钮
    │       └── 可选：从本项目参考自动补全和 Lint
    │
    └── 觉得编辑器不够好？
        └── 临时用 MarkText 写（免费可商用，桌面体验好）
            └── 但发布还是需要处理
```

1. **博客管理后台** → 给 Hexo 加 Decap CMS（成熟方案，2 小时搞定）
2. **Inbox 改进** → 加 GitHub API 发布按钮 + 参考本项目自动补全/Lint
3. **这个学生项目** → 不直接用，只摘思想。它的自动补全和 Lint 两个纯函数模块是真正有价值可复用的部分。许可证问题需要先联系作者授权
